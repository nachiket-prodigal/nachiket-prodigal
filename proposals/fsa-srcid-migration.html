srcId Migration

One tenant, many subtypes — srcId selects behaviour, not tenant identity.

FSA · Klarna / Afterpay / Amex precedent: TSI srcId 2026-07-21 · draft
Today the subtype lives in the tenant name — fsa_use2_000, fsa_use2_0ap, fsa_use1_amx, fsa_use1_aut, fsa_use2_amx — overloading it with product, region, subtype, database and environment. This moves the subtype onto srcId: one tenant fsa_use2_000 serves them all.
Principle. srcId selects provider/adapter behaviour — the code path, credentials, API endpoint and target database. It does not change tenant identity. Databases don't move: (tenant, srcId) resolves to the same physical DB the record uses today ⇒ zero data migration.

Main changes

  1. Merge the FSA tenants into one. fsa_use2_000, fsa_use2_0ap, fsa_use1_amx and fsa_use2_amx collapse into a single fsa_use2_000 override with a subtypes map keyed by srcId plus a defaultSrcId; apiBackend is dropped. config
  2. Split server.config.fsa-amex.json. Extract UAT into a new server.config.fsa-amex-uat.json so prod-amex (fsa_use1_amx) and UAT (fsa_use2_amx) resolve to the right database and cluster once both become fsa_use2_000 + amex. config
  3. Resolve the DB from (tenant, srcId). getTenantDb(tenant) → resolveDb(tenant, srcId); new db = old db, so zero data migration. TENANT_NAMES stays as the srcId-less fallback. db
  4. Thread srcId through every operation. Add it (shared SrcIdProperty) to the consumer + payment-method DTOs, and persist it on every record so scheduler / webhook / status-sync can re-resolve. api
  5. One immutable adapter per (tenant, srcId). The factory builds them from the subtypes config; getAdapter(tenant, provider, srcId?) selects one; mapSrcIdToFsaBackend maps srcId → backend. factory
  6. (Optional, separate PR) FSA adapter strategy refactor. Collapse the 17 apiBackend branches into klarna/afterpay/amex strategy files, thinning the 85 KB adapter. cleanup

Five impact areas

01

Request / response

  • srcId already on payment DTOs; add it to consumer + payment-method DTOs (distinct Repay merchant per subtype ⇒ whole lifecycle is srcId-pinned).
  • tenantId unchanged for callers: send fsa_use2_000 + srcId=amex instead of fsa_use1_amx.
  • srcId persisted on every record so scheduler / webhook / status-sync can re-resolve.
  • Absent srcId ⇒ tenant default ⇒ today's behaviour.
02

DB resolution

  • resolveDb(tenant, srcId) replaces TENANT_NAMES[tenant] alone.
  • Returns the DB the record already uses ⇒ zero migration, no residency change.
  • The amx/aut database split survives — two srcIds, two DBs.
  • TENANT_NAMES stays as the srcId-less fallback.
03

Config file changes

  • Provider block gains subtypes (keyed by srcId) + defaultSrcId; each carries database, creds, endpoints.
  • The subtypes key is the backend ⇒ drop apiBackend.
  • extractConfigValues needs no change; add subtypes to the schema.
  • Legacy flat config = one implicit subtype ⇒ every other tenant untouched.
04

Config separation (envs)

  • Three deployments, own cluster/redis each: prod, fsa-amex, fsa-amex-uat.
  • Split UAT into server.config.fsa-amex-uat.json so fsa_use2_000+amex resolves to the right DB per env.
  • Each env's subtypes list is its srcId allowlist.
  • Amex never reaches prod ⇒ no cross-region secret copy, no cutover.
05

Code org — (provider, srcId) as an adapter type

  • One resolveRoute(tenant, provider, srcId?) → {adapter, db, config}; dispatch + persistence both consume it.
  • Factory pre-instantiates one immutable adapter per (tenant, srcId) — construction-time config invariant preserved.
  • mapSrcIdToFsaBackend mirrors mapSrcIdToPayerAddress.
  • Collapsing FSA's 17 apiBackend branches into strategies = optional, separate cleanup.
OPEN

amex-aut

  • fsa_use1_aut is byte-identical to fsa_use1_amx — same Amex merchant, same creds. Only its DB name differs (a separate database, PR #151).
  • No live data ⇒ drop it. Live data ⇒ srcId=amex-aut, a behavioural alias of amex pointing at fsa_use1_aut.
  • A data question, not a code question. Kept open.
