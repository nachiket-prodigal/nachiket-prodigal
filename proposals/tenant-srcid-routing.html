Routing on the (tenant, srcId) composite key

A provider-agnostic model: the database, config, and adapter are chosen from (tenantId, srcId) rather than tenantId alone. srcId is optional — most providers never set it. How much varies per srcId differs by provider; FSA is the maximal case. Pick a provider to see what its srcId actually switches.

Why FSA is the hard case — peculiarities the generic model must absorb
✦Distinct Repay vault per srcId. FSA tokenizes via Repay, and each subtype has its own Repay merchant. A customer/token minted under one srcId is invalid on another → srcId must span the whole consumer→token→charge lifecycle, not just the charge. (TSI has no vault; generic providers use one.)
✦Separate deployment & region. Amex runs in its own pod (server.config.fsa-amex.json) in use1; FSA/Afterpay are prod use2. Serving amex from fsa_use2_000 needs FSA_AMEX_* secrets provisioned into prod.
✦Per-srcId databases, kept separate. Unlike TSI (both srcIds share one DB), FSA subtypes each keep their own book — including fsa_use1_amx vs fsa_use1_aut, an intentional same-creds split (PR #151). The composite key preserves them with zero migration.
✦Different backend code paths. srcId selects an entirely different strategy (FSA vs Amex ProAgent vs Afterpay), not just a credential set. UAT (fsa_use2_amx) is an environment, not a subtype — it should not become a prod srcId.
The five decisions (generic)
1 · Config
Per-(tenant, srcId) blocks
paymentProviders.<p>.subtypes.<srcId>, each with its own creds + optional database. Flat legacy config = one implicit default srcId.
2 · Data
Per-provider choice
DB defaults to TENANT_NAMES[tenant]; a subtype may override it. TSI shares; FSA keeps books separate → no migration.
3 · API
srcId optional, everywhere
On every op (incl. consumer/token where a vault is subtype-scoped), persisted on every record. tenantId semantics unchanged.
4 · Runtime
Widen the registry key
Immutable adapter per (tenant, srcId), selected at request time. Stores take the resolved route, not a bare tenantId.
5 · Quality
One resolver, one path
request → resolveRoute → { adapter, db }. No srcId branching scattered through adapters or stores.
✓ Live proof
TSI already does it
tsi_use2_000 + tsi_use2_sme share one DB, split by srcId (NTTA/SME). This generalises the pattern.
