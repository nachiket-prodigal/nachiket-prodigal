How to read this doc

Purpose. This is a decision aid for splitting the FSA adapter into separate adapter classes per backend. Today, fsa.adapter.ts (~2200 lines) bundles three backends behind an apiBackend config flag: FSA (Klarna), Amex (ProAgent), and Afterpay. The structural question is how to factor them apart — inheritance or composition.

How to read it.

  1. Use the Approach toggle in the sidebar to switch between Inheritance and Composition. The file layout, class diagram, and code skeleton all update together.
  2. Pick a Future scenario to see how each approach copes when requirements change. Verdicts (✓ / ~ / ✗) sit side by side.
  3. The Tradeoff matrix at the bottom scores both approaches across 10 dimensions.
  4. The Implementation prompt at the bottom-left of the sidebar updates live; copy it to relay the decision into an implementation conversation.

Scope. This doc is about structural choice only. Rollout sequencing — how to ship the refactor without breaking existing tenants or persisted data — is intentionally kept in a separate doc: fsa-split-rollout-phases.md.

Current recommendation. Composition. The amex+afterpay overlap (shared auth, inferOfferType, ROUND_DOWN) fits naturally as shared utility modules + three sibling adapters, and avoids the awkward 3-deep "FSAProAgent IS-A FSA" intermediate base that inheritance would require. The toggle stays in place so the reasoning is auditable.
